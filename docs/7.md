# **Выполнение задания. Проектная документация**

## **Цель работы**

Целью выполнения задания является формирование у обучающегося практических навыков **проектирования информационных систем** и подготовки **комплекта проектной документации**, отражающего полный цикл предварительного проектирования программного продукта.

В рамках задания обучающийся должен:

* проанализировать выбранную предметную область;
* описать назначение и границы будущей информационной системы;
* определить пользователей, сущности и ключевые бизнес-процессы;
* разработать архитектурное решение системы;
* спроектировать структуру базы данных;
* спланировать процесс разработки программного продукта;
* представить план разработки в виде диаграммы Ганта;
* выполнить оптимизацию пользовательского интерфейса на основе прототипа.

Результатом работы является **структурированный репозиторий проекта**, содержащий взаимосвязанные документы, схемы и материалы, оформленные в едином стиле и отражающие одну логически целостную информационную систему.

> Шаблон отчета о выполнении содержит скрины со всеми пунктами и описание для пунктов, которые тут не представлены.
В конце вы даете ссылку на свой репозиторий.

---

## **Общее задание**

Обучающемуся необходимо разработать **комплект проектной документации по выбранной предметной области**, используя:

* структуру репозитория;
* принципы оформления документации;
* стиль описания;
* подходы к моделированию,

реализованные в **тестовом проекте**, выполненном на предыдущей практике.

---

## **В ходе выполнения задания обучающийся должен**

* выполнить анализ предметной области;
* определить цели и назначение разрабатываемой системы;
* описать участников и их роли;
* построить архитектурную схему системы;
* спроектировать структуру базы данных;
* выполнить логическую и концептуальную проработку данных;
* спланировать процесс разработки в виде диаграммы Ганта;
* оформить итоговый комплект проектных материалов.

---

## **1. Подготовительный этап**

### **1.1. Изучение тестового проекта**

На данном этапе обучающийся использует проект, выполненный на предыдущей практике, и выполняет подготовительные действия для создания нового репозитория.

#### Требуется создать новый проект для размещения на GitHub, включающий:

* каталог `/docs` — проектная и пояснительная документация;
* каталог `/src` — приложение на Flask (по материалам предыдущей практики);
* каталог `/db` — материалы, связанные с базой данных;
* файл `README.md` — общее описание проекта.

#### При изучении тестового проекта необходимо обратить внимание на:

* логическую структуру проектной документации;
* оформление файлов в формате Markdown;
* использование диаграмм (Mermaid, ERD, flowchart);
* способы описания предметной области;
* согласованность документов между собой;
* общую организацию репозитория.

Результатом этапа является **понимание структуры будущего проекта**, которая будет воспроизведена для новой предметной области.

---

### **1.2. Определение предметной области**

Обучающийся выбирает и закрепляет за собой предметную область.

#### Примеры предметных областей:

* туроператор (формирование и продажа турпакетов);
* онлайн-турагентство (OTA);
* система бронирования авиабилетов;
* пекарня или кондитерская;
* учёт заявок на услуги;
* сервис приёма обращений клиентов.

#### Требования к предметной области:

* предметная область должна быть **ограниченной по объёму**;
* должна содержать **несколько сущностей**, связанных между собой;
* должна включать **различные роли пользователей**;
* должна содержать **несколько бизнес-процессов**, подлежащих автоматизации.

## **2. Разработка проектной документации**

### **2.1. Сбор информации**

Перед оформлением документации необходимо ответить на следующие вопросы:

* кто является пользователями системы;
* какие роли существуют в системе;
* какие действия выполняют пользователи;
* какие объекты и данные присутствуют в системе;
* какие процессы выполняются вручную и подлежат автоматизации;
* какие задачи должна решать информационная система.

Ответы на данные вопросы используются как основа для дальнейшего проектирования.

---

### **2.2. Оформление описания предметной области**

Описание предметной области оформляется в виде отдельного документа и должно быть **структурированным, логичным и однозначным**.

#### Рекомендуемая структура файла `docs/domain.md`

**Назначение документа**
Фиксирует, **что автоматизируется**, **кто участвует** и **какие процессы существуют**.

#### Пример содержимого `docs/domain.md`

```markdown
# 1. Предметная область

## 1.1. Краткое описание
Предметная область — учёт заявок на ремонт и обслуживание оборудования
в организации.

Информационная система предназначена для автоматизации процесса
приёма, обработки и контроля выполнения заявок.

## 1.2. Основные участники
- Пользователь — создаёт заявки и отслеживает их статус.
- Исполнитель — принимает заявки в работу.
- Администратор — управляет пользователями и справочниками.

## 1.3. Основные процессы
1. Создание заявки.
2. Назначение исполнителя.
3. Выполнение заявки.
4. Закрытие заявки.
```

Описание должно быть выполнено **в нейтральном техническом стиле**, без привязки к конкретной реализации.

---

### **2.3. Разработка архитектурной схемы**

#### **2.3.1. Определение компонентов системы**

На основе анализа предметной области необходимо определить основные компоненты системы:

* пользовательский интерфейс (Web, Desktop, Mobile);
* серверная часть (Backend API);
* модуль бизнес-логики;
* база данных;
* внешние сервисы (при необходимости).

---

#### **2.3.2. Построение архитектурной схемы**

Архитектурная схема должна отражать:

* состав компонентов системы;
* взаимодействие между компонентами;
* направления потоков данных.

Допускается использование диаграмм в нотации Mermaid.

---

#### **2.3.3. Размещение схемы в документации**

##### Файл `docs/architecture.md`

**Назначение документа**
Показывает, **из каких компонентов состоит система** и **как они взаимодействуют**.

##### Пример содержимого

``` text
## 2. Архитектура системы

### 2.1. Общая схема архитектуры

```mermaid

    UI --> API
    API --> Logic
    Logic --> DB
```

### **2.4. Проектирование структуры базы данных**

### **2.4.1. Определение сущностей**

На основе предметной области необходимо определить основные сущности системы и их назначение.

---

### **2.4.2. Построение ER-диаграммы**

ER-диаграмма должна отражать:

- сущности;
- их атрибуты;
- первичные и внешние ключи;
- типы связей между сущностями.

### **2.4.3. Оценка необходимости оптимизации структуры БД**

Необходимо оценить:

- необходимость индексации;
- целесообразность нормализации.

---

### **2.4.4. Вынос структуры БД в отдельные файлы**

- `db/schema.sql` — SQL-скрипт создания таблиц;
- `docs/database.md` — описание структуры БД.


#### Рекомендуемая структура файла `docs/database.md` — структура базы данных

##### Назначение документа  
Описывает **модель данных**, сущности и связи между ними.

##### **Пример содержимого `docs/database.md`:**

```markdown
# 3. Структура базы данных

## 3.1. Назначение базы данных
База данных предназначена для хранения информации
о пользователях, заявках и статусах их выполнения.

## 3.2. Основные сущности

- User — пользователь системы;
- Request — заявка;
- Status — статус заявки.

## 3.3. ER-диаграмма

```mermaid
erDiagram
    USER {
        int id PK
        string name
        string email
    }

    REQUEST {
        int id PK
        string title
        string description
        int user_id FK
        int status_id FK
    }

    STATUS {
        int id PK
        string title
    }

    USER ||--o{ REQUEST : "создаёт"
    STATUS ||--o{ REQUEST : "определяет"

## 3.4. Особенности проектирования

Для ускорения поиска заявок по пользователю
предусматривается индекс по полю `user_id`.
```

## **4. `db/schema.sql` — SQL-скрипт**

### Назначение документа  
Формально описывает **структуру БД в виде SQL**.

### **Пример `db/schema.sql`:**

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL
);

CREATE TABLE statuses (
    id SERIAL PRIMARY KEY,
    title VARCHAR(50) NOT NULL
);

CREATE TABLE requests (
    id SERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    description TEXT,
    user_id INT REFERENCES users(id),
    status_id INT REFERENCES statuses(id)
);
```

## **6. Файл `README.md`— описание проекта**

### Назначение документа

Кратко объясняет, **что это за система**, для чего она предназначена и из каких частей состоит.

### **Пример содержимого `README.md`:**

```markdown
# ServiceDesk Lite

Учебный проект информационной системы для учёта и обработки заявок пользователей.

## Назначение системы
Система предназначена для приёма заявок от пользователей,
назначения ответственных исполнителей,
отслеживания статусов выполнения
и формирования отчётов по обращениям.

## Основные функции
- создание заявок;
- изменение статусов заявок;
- назначение ответственных;
- хранение истории обращений.

## Структура репозитория
- `/docs` — проектная документация;
- `/src` — исходный код приложения;
- `/db` — структура базы данных.

Проект выполнен в рамках прозводственной практики.
```
